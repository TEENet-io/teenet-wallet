# 签名与 TEE 信任模型

本页说明当钱包发送签名请求时，**TEENet service 内部**发生了什么。钱包本身从不接触私钥 -- 所有密码学操作均通过 SDK 委托给 TEENet service。

---

## 门限密码学

TEENet 使用门限密码学消除单点泄露风险：

- 密钥生成时，私钥被**分片**到 3--5 个 TEE 节点。每个节点持有私钥的一个分片。
- 签名需要 **M-of-N** 协作（例如 3 个节点中的 2 个，或 5 个节点中的 3 个）。单个被攻破的节点无法生成有效签名。
- 完整私钥**从未被还原** -- 不在任何节点上，不在内存中，签名过程中也不会出现。每个节点使用自己的分片计算部分签名，协调者从部分结果组装最终签名。

这意味着攻击者需要同时攻破多个 TEE 飞地，才能提取足够的分片来重建密钥。

---

## 签名流程

当钱包调用 `sdk.Sign(ctx, msgBytes, keyName)` 时，TEENet service 内部的流程如下：

```
Wallet                    TEENet Service
  │                           │
  │  Sign(msg, keyName)       │
  │ ─────────────────────────>│
  │                           │
  │              ┌──────┐ ┌──────┐ ┌──────┐
  │              │ TEE  │ │ TEE  │ │ TEE  │
  │              │ Node │ │ Node │ │ Node │
  │              │  1   │ │  2   │ │  3   │
  │              └──┬───┘ └──┬───┘ └──┬───┘
  │                 │        │        │
  │                 │ partial signatures
  │                 │        │        │
  │                 └───┬────┘────┘
  │                     ▼
  │              Initiating node
  │              assembles final
  │              signature
  │                     │
  │  Signature          │
  │ <───────────────────┘
  │
```

1. 钱包通过 SDK 将消息字节和密钥名称发送给 TEENet service。
2. 其中一个 TEE 节点发起签名流程，并将请求分发给持有该密钥分片的其他节点。
3. 每个参与的 TEE 节点在飞地内使用自己的密钥分片计算**部分签名**。
4. 发起节点收集所需数量的部分签名（N 个中的 M 个），组装最终的门限签名。
5. 完整签名返回给钱包，钱包将已签名交易广播到区块链。

## 什么是 `app_instance_id`？

每个在 TEENet service 注册的应用都会获得一个**应用实例 ID**。此 ID 有两个用途：

- **身份标识**：告知 TEENet service 哪个应用在发起请求，实现应用间的密钥隔离。
- **签名模式**：该 ID 决定签名请求的处理方式。

钱包在启动时从环境中读取 `APP_INSTANCE_ID`，并通过 `SetDefaultAppInstanceIDFromEnv()` 传递给 SDK。后续所有密钥生成和签名调用都使用此 ID。

### 签名模式

| 模式 | 行为 |
|------|------|
| **Direct** | 协调者立即签名并返回签名结果。用于标准钱包操作。 |
| **Multi-instance voting** | 多个实例并行运行。签名请求进入待处理状态，直到足够多的实例独立提交相同请求达到阈值。用于任何单一实例都不应独自签名的多方协作场景。 |

在典型的钱包部署中，应用使用 **Direct** 模式。钱包自身的审批系统（阈值策略、每日限额）在应用层处理人工授权，签名请求到达 TEENet service 之前已完成审批。
