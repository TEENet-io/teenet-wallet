# TEENet Wallet -- 技术白皮书

**版本 0.2.0 | 2026 年 3 月**

---

## 摘要

TEENet Wallet 是一款多链加密货币钱包，通过在可信执行环境 (TEE) 节点集群中使用门限密码学分布式管理私钥材料，消除了传统密钥管理中固有的单点故障问题。任何单台机器都不会持有完整的私钥。签名操作需要 M-of-N 个 TEE 节点通过 FROST（用于 Ed25519/Schnorr 曲线）或 GG20（用于 ECDSA/secp256k1 曲线）协同完成，在保持实际交易延迟可控的同时提供拜占庭容错能力。钱包对外暴露的 REST API 支持双重消费模式：AI 代理使用 API Key 进行程序化自动操作，人类操作员使用 WebAuthn/Passkey 硬件凭证执行敏感操作。分层安全模型包括合约白名单、方法级别限制、基于实时价格预言机的美元计价审批阈值，以及具有授权/捕获回滚语义的每日支出限额，为防范凭证泄露和软件漏洞提供纵深防御。系统支持 Ethereum 及所有 EVM 兼容链，以及 Solana，并可在运行时动态扩展额外的 EVM 网络。

## 1. 引言

### 1.1 密钥管理难题

加密货币安全归结为一个核心挑战：保护私钥。现有解决方案呈现出一系列不尽如人意的权衡取舍：

- **热钱包**（MetaMask、浏览器内嵌钱包）将密钥存储在通用硬件上的软件内存或加密文件中。它们使用便捷，但容易受到恶意软件、钓鱼攻击和供应链攻击的威胁。单台机器被攻破就意味着资金的全部损失。

- **硬件钱包**（Ledger、Trezor）将密钥隔离在安全元件中，但每次签名都需要物理人工交互。它们不适用于自动化工作流、DeFi 策略或 AI 代理操作。同时它们仍然是单一设备——如果没有妥善管理助记词，设备丢失或损坏可能意味着密钥永久丢失。

- **MPC 钱包**（Fireblocks、Fordefi）将密钥份额分布在多方之间，消除了单点故障。然而，商业 MPC 解决方案通常是不透明的、专有的且价格昂贵的。密钥份额驻留在供应商的基础设施中，产生了托管信任假设。

- **智能合约钱包**（Safe、ERC-4337 Account Abstraction）将授权逻辑放在链上。它们是特定链的，配置更改需要消耗 gas，且无法为链下协议签署任意消息。

### 1.2 TEENet Wallet 的方案

TEENet Wallet 利用分布式 TEE-DAO 集群处理所有密码学操作，将硬件隔离的安全特性与软件钱包的操作灵活性相结合：

1. **门限密码学** —— 私钥以分布式形式生成和使用。完整密钥永远不会在任何单台机器上被还原。

2. **TEE 硬件隔离** —— 密钥份额驻留在可信执行环境内部，提供硬件级内存加密和远程证明。

3. **双重认证** —— API Key 支持 AI 代理自动化操作，WebAuthn/Passkey 凭证则通过硬件支持的人类在场验证为敏感操作把关。

4. **纵深防御** —— 合约白名单、方法限制、美元审批阈值、每日支出限额和 CSRF 防护构成多个独立的安全层。

5. **多链支持** —— 单一部署即可处理 Ethereum、Solana 及任何 EVM 兼容链，并支持运行时添加新链。

### 2.2 数据流

**钱包创建 (DKG)：**

1. 客户端发送 `POST /api/wallets` 请求并指定链选择。
2. TEENet Wallet 验证请求并确定所需的密码学协议（EVM 使用 ECDSA，Solana 使用 Schnorr）。
3. TEENet SDK 在 TEE-DAO 集群上发起分布式密钥生成。对于 ECDSA (GG20)，这涉及多轮通信，可能需要 60-120 秒。
4. 每个 TEE 节点生成并存储其密钥份额。公钥经聚合后返回。
5. TEENet Wallet 从公钥派生区块链地址（EVM 使用 Keccak-256 截断，Solana 使用 Ed25519 编码），并持久化钱包记录。

**交易签名：**

1. 客户端发送转账或合约调用请求。
2. TEENet Wallet 在服务端构建未签名交易（EVM 使用 EIP-1559，Solana 使用 Solana 交易消息）。
3. 审批策略引擎评估该交易是否需要人工审批（基于美元金额、方法风险等级、自动审批设置）。
4. 如果已审批（或低于阈值），签名哈希通过 TEENet SDK 发送至 TEENet 服务。
5. TEENet 服务在 TEE-DAO 集群上协调 M-of-N 门限签名。
6. 签名返回给 TEENet Wallet，后者组装已签名交易并广播至区块链 RPC 端点。
7. 交易哈希返回给客户端。

### 2.3 数据库设计

TEENet Wallet 使用 SQLite 的 WAL（预写日志）模式以获得并发读取性能和运维简便性。数据库模式包含七张表：

| 表名 | 用途 |
|-------|---------|
| `users` | 关联至 UMS Passkey 身份的本地用户账户 |
| `wallets` | TEE 支持的钱包，包含链、地址、密钥引用和状态 |
| `allowed_contracts` | 每个钱包的合约白名单及方法限制 |
| `approval_policies` | 每个钱包的美元阈值和每日限额配置 |
| `approval_requests` | 待处理/已完成的审批队列及交易上下文 |
| `audit_logs` | 不可变的操作日志，包含认证模式、IP 和详情 |
| `idempotency_records` | 转账去重缓存 |
| `custom_chains` | 用户添加的 EVM 链配置 |

## 3. 密码学设计

### 3.1 门限密码学

TEENet Wallet 依赖两种门限签名方案，根据目标区块链的需求自动选择：

**FROST (Flexible Round-Optimized Schnorr Threshold Signatures)：**
用于 Ed25519 曲线（Solana）。FROST 提供两轮签名协议，速度显著快于 GG20。该方案生成标准 Ed25519 签名，在链上与单签名者签名无法区分。

**GG20 (Gennaro-Goldfeder 2020)：**
用于 secp256k1 上的 ECDSA（Ethereum 及所有 EVM 链）和 secp256r1（NIST P-256，用于 WebAuthn）。GG20 在无可信分发者的情况下实现分布式 ECDSA 签名。该协议需要预计算参数（安全素数、Paillier 密钥），由 prime-service 生成以避免阻塞签名操作。

| 协议 | 曲线 | 区块链 | DKG 时间 | 签名时间 |
|----------|-------|-----------|----------|-----------|
| FROST | Ed25519 | Solana | ~5-10s | <1s |
| GG20 | secp256k1 | Ethereum, EVM | 60-120s | 2-5s |
| GG20 | secp256r1 | WebAuthn（内部） | 60-120s | 2-5s |

### 3.2 分布式密钥生成

密钥生成由所有 TEE-DAO 节点使用可验证秘密共享协议协作完成。该过程确保：

1. **无可信分发者** —— 没有单一方选择私钥。
2. **可验证性** —— 每个节点可通过零知识证明验证其他节点提供的份额是否有效（FROST 使用 Feldman VSS，GG20 使用 Pedersen 承诺）。
3. **门限属性** —— 任意 M 个节点（N 个节点中的子集）可以重构有效签名，但少于 M 个节点无法获取任何关于私钥的信息。

典型配置为 3-of-5：五个 TEE 节点参与密钥生成，任意三个节点可协作签名。这可以容忍最多两个节点的故障或被攻破。

### 3.3 M-of-N 签名

签名流程由 TEENet 服务协调，其充当会话协调者：

1. 协调者接收待签名的消息哈希。
2. 从集群中选择 M 个可用的 TEE 节点。
3. 每个被选中的节点使用其密钥份额执行本地计算。
4. 部分签名被组合成完整的门限签名。
5. 生成的签名是标准的 ECDSA 或 Schnorr 签名，与单签名者签名无法区分。

M-of-N 属性意味着即使攻击者攻破了 M-1 个 TEE 节点，仍然无法生成有效签名。结合 TEE 硬件隔离，这提供了强大的安全保障。

### 3.4 密钥份额隔离

每个 TEE 节点将其密钥份额存储在可信执行环境内部。TEE 技术（Intel SGX、AMD SEV、ARM TrustZone）提供：

- **内存加密** —— 密钥材料在 RAM 中加密存储，即使宿主操作系统或虚拟机管理程序也无法访问。
- **远程证明** —— 客户端可在发送敏感数据前，以密码学方式验证 TEE 内部运行的是正确的软件。
- **密封存储** —— 密钥份额被加密绑定到特定的 TEE 实例，无法提取到其他硬件上运行。

## 4. 安全模型

### 4.1 威胁模型

TEENet Wallet 旨在抵御以下威胁：

| 威胁 | 缓解措施 |
|--------|------------|
| API Key 被盗 | API Key 无法删除钱包、移除合约、更改策略或审批高风险交易。所有此类操作都需要 Passkey。 |
| Passkey 会话令牌被盗 | 审批、删除和 API Key 生成需要*新鲜的* WebAuthn 断言（硬件密钥交互），而不仅仅是会话令牌。 |
| 单个 TEE 节点被攻破 | M-of-N 门限签名确保少于 M 个被攻破的节点无法生成签名。 |
| 恶意合约交互 | 三层合约安全：白名单、方法限制、高风险方法门控。 |
| 价格操纵绕过阈值 | 稳定币价格硬编码为 $1；波动资产价格使用 CoinGecko 并设有 10 秒缓存。DeFi 调用的 `amount_usd` 字段提供额外的调用方报告值，系统取计算值和报告值中的较大者。 |
| 重放攻击（重复交易） | Idempotency-Key 头部防止重复转账。EVM nonce 管理防止链上重放。 |
| CSRF 攻击浏览器会话 | 所有 Passkey 会话的状态变更请求都需要 X-CSRF-Token 头部。 |
| DKG 资源耗尽 | 钱包创建（按用户）和注册（按 IP）的速率限制防止 TEE 集群滥用。 |

### 4.2 认证层级

TEENet Wallet 实现了双重认证架构，专为自动化代理与人类操作员共存场景设计：

**API Key 认证（`ocw_` 前缀）：**
- 通过 Passkey 认证端点生成（需要新鲜的 WebAuthn 断言）。
- 以加盐 SHA-256 哈希存储；原始密钥仅在生成时展示一次。
- 适用于 AI 代理、后端服务和自动化交易机器人。
- 无法执行破坏性操作（钱包删除、合约移除、策略删除、账户删除）。
- 无法审批待处理的审批请求。
- 受每个密钥的速率限制约束（可配置，默认 200 请求/分钟）。

**Passkey 会话认证（`ps_` 前缀）：**
- 通过 WebAuthn 登录仪式创建（硬件密钥交互）。
- 会话在 24 小时后过期。
- 所有状态变更请求都需要 X-CSRF-Token 头部。
- 敏感操作（审批、删除、密钥生成）额外需要在同一请求体中提供*新鲜的* WebAuthn 断言。

这种"新鲜凭证"要求是一项关键安全特性：即使攻击者获取了有效的会话令牌（例如通过 XSS），如果没有对用户硬件密钥的物理访问权限，也无法审批高价值交易。

### 4.3 合约白名单

智能合约交互经过两个安全层：

**第一层 —— 地址白名单：**
只有预先批准的合约地址（EVM）或程序 ID（Solana）才能被调用。与未列入白名单的地址的任何交互都会被拒绝，返回 HTTP 403。将合约添加到白名单需要 Passkey 认证（如果由 API Key 发起则创建待审批请求）。

**第二层 —— API Key 合约调用强制审批：**
所有通过 API Key 发起的合约调用都需要 Passkey 审批。这是一个审慎的设计决策：合约交互的风险高于简单转账，钱包对每一笔由 Agent 发起的合约调用强制要求人工确认。通过 Passkey 会话（人类已在场）发起的合约调用则立即执行。

便捷端点 `approve-token` 和 `revoke-approval` 同样始终需要 Passkey 审批，因为它们涉及授权或撤销第三方支出权限。

### 4.4 审批策略

每个钱包可以设置一个以美元计价的审批策略，包含两个可配置的阈值：

**单笔交易阈值 (`threshold_usd`)：**
当转账或合约调用涉及的美元金额超过此阈值时，请求将作为待审批项（HTTP 202）排队，而不是立即执行。用户必须通过 Web UI 使用新鲜的 Passkey 断言进行审批。

**每日支出限额 (`daily_limit_usd`)：**
可选的 UTC 日总美元支出硬性上限。与阈值（仅触发审批）不同，每日限额是硬性阻断——一旦达到，无论认证模式如何，都不允许进一步转账。

**价格转换：**
美元金额在请求时使用实时价格计算：
- 波动资产（ETH、SOL）：CoinGecko API，10 秒缓存。
- 稳定币（USDC、USDT、DAI 等）：硬编码为 $1.00。
- DeFi 调用：可选的 `amount_usd` 字段允许调用方报告复杂操作的美元价值（例如 Uniswap 兑换）。系统取原生资产计算值和调用方报告值中的较大者。

**授权/捕获模式：**
每日支出追踪使用预扣除模型并支持回滚：

1. 签名前，预估美元金额从每日预算中扣除。
2. 如果签名或广播失败，扣除金额将被回滚。
3. 如果交易需要审批（超过阈值），预扣除将立即回滚；在审批执行时将重新扣除。

这可以防止失败交易的"虚假支出"，同时确保在并发请求期间每日限额不会被无声超出。每个钱包的互斥锁序列化检查和扣除操作，以防止 TOCTOU 竞态条件。

### 4.5 CSRF 和速率限制

**CSRF 防护：**
来自 Passkey 会话的所有状态变更 API 请求（POST、PUT、PATCH、DELETE）必须包含与登录时返回的令牌匹配的 `X-CSRF-Token` 头部。这可以防止针对基于浏览器的 Web UI 的跨站请求伪造攻击。

**速率限制：**
三个独立的速率限制器保护不同的资源：
- 通用 API 速率限制：按 API Key，可配置（默认 200 请求/分钟）。
- 钱包创建速率限制：按用户，可配置（默认 5 次/分钟）。这一点至关重要，因为每次钱包创建都会触发消耗大量集群计算资源的 TEE DKG 操作。
- 注册速率限制：按 IP，可配置（默认 10 次/分钟）。防止对公开注册端点的匿名滥用。

## 5. 多链支持

### 5.1 EVM 链（Ethereum、L2、测试网）

所有 EVM 链使用 secp256k1 上的 ECDSA。TEENet Wallet 构建具有动态费用估算的 EIP-1559（type-2）交易：

- 基础费用和优先费用在构建时从链 RPC 查询。
- Gas 限制通过 `eth_estimateGas` 估算。
- 内置 nonce 管理器按地址跟踪待处理的 nonce，以支持并发交易提交而不发生 nonce 冲突。
- 交易使用 TEE 提供的 ECDSA 签名组装，并通过 `eth_sendRawTransaction` 广播。

**支持的内置链：**
Ethereum Mainnet、Optimism Mainnet、Sepolia Testnet、Holesky Testnet、Base Sepolia Testnet、BSC Testnet。

**自定义链扩展：**
可在运行时通过 `POST /api/chains`（需要 Passkey）添加额外的 EVM 链，指定名称、RPC URL、货币符号和链 ID。自定义链持久化在数据库中，重启后仍然有效。

### 5.2 Solana

Solana 使用通过 FROST 门限方案生成的 Ed25519 签名。TEENet Wallet 支持：

- **原生 SOL 转账** —— 标准系统程序转账。
- **SPL 代币转账** —— TransferChecked 指令，对尚未拥有关联代币账户 (ATA) 的接收方自动创建 ATA。
- **包装/解包装 SOL** —— 为 DeFi 互操作性创建和关闭 wSOL ATA。
- **通用程序调用** —— 使用调用方指定的账户列表和指令数据的任意 Solana 程序指令。

Solana 交易在审批时使用新鲜的 blockhash 重新构建（因为 blockhash 大约在 60 秒后过期），确保已审批的交易仍然有效。

### 5.3 地址派生

地址派生在服务端从 TEE 生成的公钥完成：

- **EVM：** 对未压缩公钥（不含 0x04 前缀字节）进行 Keccak-256 哈希，取最后 20 字节，使用 EIP-55 混合大小写校验编码。
- **Solana：** Ed25519 公钥直接用作地址，以 Base58 编码。

## 6. AI 代理集成

### 6.1 面向代理安全的双重认证设计

TEENet Wallet 的认证模型专为 AI 代理管理加密货币这一新兴模式设计：

1. **API Key 授予操作访问权限** —— AI 代理可以创建钱包、查询余额、在阈值以下转账代币，以及与自动审批合约交互。

2. **Passkey 控制不可逆操作** —— 人类所有者保留对钱包删除、合约白名单管理、策略更改和高价值交易审批的独占控制权。

3. **审批队列弥合差距** —— 当代理发起的操作超出其权限（例如超过美元阈值的转账）时，请求会排队等待人工审核，而不是直接拒绝。代理会收到 `approval_id` 和 `approval_url`，可以将其展示给用户。

这种设计使 AI 代理能够在明确定义的边界内自主运行，同时对安全关键决策保留人类监督。

### 6.2 DeFi 调用的美元报告

`contract-call` 端点上的 `amount_usd` 字段解决了 DeFi 中的一个根本挑战：交易的链上价值通常是不透明的。

例如，Uniswap 兑换将交易参数编码在 calldata 中，但发送的原生 ETH 值可能为零（对于代币到代币的兑换）。如果没有 `amount_usd`，审批策略引擎无法知道正在执行的是 $10 还是 $10,000 的兑换。

通过接受调用方报告的美元值，TEENet Wallet 使 AI 代理能够为复杂的 DeFi 操作提供自己的估值。系统取原生资产计算值和报告的 `amount_usd` 中的较大者，防止代理通过低报来绕过阈值。

### 6.3 OpenClaw 技能系统

TEENet Wallet 包含一个 OpenClaw 技能定义（`skill/tee-wallet/`），通过 AI 助手平台实现与钱包的自然语言交互。该技能通过结构化接口暴露钱包管理、余额查询和交易操作，供 AI 代理调用。

## 7. 运维考量

### 7.1 部署

TEENet Wallet 以静态编译的 Go 二进制文件或 Docker 镜像形式分发：

- **Docker 镜像**使用多阶段构建，运行时使用非 root 用户。
- 唯一的运行时依赖是一个在配置的 `SERVICE_URL` 上可达的 TEENet 网格节点。
- 所有配置通过环境变量完成（12-factor 应用风格）。
- 服务器默认绑定到 `0.0.0.0:8080`。

### 7.2 数据库

SQLite 的 WAL 模式提供：
- **并发读取** —— WAL 模式允许多个读取者而不阻塞写入者，适合读密集型工作负载模式。
- **运维简便** —— 无需管理外部数据库进程。数据库是配置的 `DATA_DIR` 中的单个文件。
- **忙等待超时** —— 配置为 5 秒，以优雅地处理并发写入竞争。
- **自动迁移** —— 模式在启动时通过 GORM AutoMigrate 创建和更新。

对于处理高并发写入的生产部署，数据目录应位于本地 SSD 存储（而非网络附加存储）上，以最小化 WAL 检查点延迟。

### 7.3 优雅关闭

服务器以 30 秒的优雅关闭周期处理 SIGINT 和 SIGTERM 信号：

1. 停止接受新连接。
2. 等待进行中的请求完成（最多 30 秒）。
3. 关闭数据库连接池。
4. 停止后台 goroutine（会话清理、速率限制器清理、幂等性清理、价格服务）。

这确保了正在进行的签名操作不会被中断，否则可能导致 nonce 处于不一致状态。

### 7.4 审计日志

每个重要操作都记录在 `audit_logs` 表中，包含：

- **用户 ID** —— 执行操作的人。
- **操作** —— 执行了什么（例如 `transfer`、`wallet_create`、`approval_approve`、`apikey_generate`）。
- **认证模式** —— 用户如何认证（`passkey` 或 `apikey`）。
- **IP 地址** —— 请求来源。
- **钱包 ID** —— 受影响的钱包（如适用）。
- **详情** —— JSON 上下文，包含交易哈希、金额、接收方和审批 ID。
- **状态** —— 结果（`success`、`pending`、`failed`）。

审计写入采用尽力而为的方式（错误被记录但不会导致请求失败），以防止审计基础设施问题阻塞关键操作。

### 7.5 安全头部

Web UI 使用严格的安全头部提供服务：

- `Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self'`
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: strict-origin-when-cross-origin`

## 8. 与替代方案的比较

### 8.1 与热钱包的比较（MetaMask、内嵌钱包）

| 维度 | 热钱包 | TEENet Wallet |
|-----------|-----------|---------------|
| 密钥存储 | 软件（浏览器、磁盘） | 分布在 TEE 硬件上 |
| 单点故障 | 是（单一设备） | 否（M-of-N 门限） |
| 自动化支持 | 有限（需要浏览器扩展） | 原生 API Key 支持 |
| 人类监督 | 无（任何软件都可签名） | Passkey 门控敏感操作 |
| 恢复 | 助记词（单一秘密） | 跨 TEE 集群的密钥重分配 |

### 8.2 与硬件钱包的比较（Ledger、Trezor）

| 维度 | 硬件钱包 | TEENet Wallet |
|-----------|----------------|---------------|
| 自动化 | 不可能（需要物理按键） | 在策略范围内完全 API 自动化 |
| 冗余 | 单一设备（通过助记词备份） | M-of-N 跨多个 TEE 节点 |
| 多链 | 依赖设备固件 | 服务端实现，运行时可扩展 |
| AI 代理集成 | 无 | 原生双重认证设计 |
| 交易审核 | 设备屏幕（有限） | Web UI 提供完整上下文 |

### 8.3 与 MPC 钱包的比较（Fireblocks、Fordefi）

| 维度 | 商业 MPC | TEENet Wallet |
|-----------|---------------|---------------|
| 基础设施 | 供应商托管（托管信任） | 自托管 TEE 集群 |
| 密钥份额位置 | 供应商 + 客户端 | 用户控制的 TEE 节点 |
| 成本 | 按交易或订阅收费 | 仅基础设施成本 |
| 透明度 | 专有协议 | 开放的门限密码学（FROST、GG20） |
| 可定制性 | 供应商定义的策略 | 完全可配置的策略和白名单 |
| AI 代理支持 | 因供应商而异 | 原生双重认证设计 |

### 8.4 与智能合约钱包的比较（Safe、Account Abstraction）

| 维度 | 智能合约钱包 | TEENet Wallet |
|-----------|----------------------|---------------|
| 链支持 | 每次部署单条链 | 单实例多链 |
| 配置变更需要 Gas | 是（链上交易） | 否（链下策略管理） |
| 任意消息签名 | 有限（EIP-1271） | 完全支持（任何消息格式） |
| Solana 支持 | 否（仅 EVM） | 是 |
| 恢复 | 链上社交恢复 | TEE 密钥重分配 |
| 延迟 | 策略需链上确认 | 即时链下策略更新 |

## 9. 未来工作

以下领域正在积极开发或考虑中：

- **Bitcoin 支持** —— 通过 FROST 在 secp256k1 上实现 BIP-340 Schnorr 签名，支持原生 Bitcoin 和 Taproot 交易。

- **密钥重分配** —— 在不生成新密钥或更改链上地址的情况下，在变更后的 TEE 节点集合之间重新分配密钥份额。底层 TEE-DAO 集群已支持重分配；计划将其集成到钱包 API 中。

- **多用户审批工作流** —— 除 M-of-N TEE 节点外，还要求 M-of-N 人类审批者批准高价值交易，适用于组织资金管理。

- **支出策略 DSL** —— 更丰富的策略语言，支持按合约限额、时间段限制、接收方白名单和速度检查。

- **硬件证明验证** —— 钱包对 TEE 证明报告的密码学验证，提供密钥份额在真实 TEE 安全区域内运行的端到端证明。

- **ERC-4337 Bundler 集成** —— 支持 Account Abstraction UserOperations，实现无 gas 交易和批量操作。

- **跨链原子操作** —— 跨多条链的协调签名，用于桥接操作和跨链 DeFi。

- **增强审计与合规** —— 可导出的审计追踪、审批 Webhook 通知，以及与合规监控服务的集成。

## 10. 结论

TEENet Wallet 证明了门限密码学和可信执行环境能够为传统加密货币密钥管理提供实用的、生产级的替代方案。通过将密钥份额分布在多个 TEE 节点上，并要求每次签名都需要 M-of-N 节点协作，系统消除了困扰热钱包的单点故障问题，同时保持了硬件钱包所缺乏的可编程性。

双重认证模型——API Key 用于代理、Passkey 用于人类——满足了在人类监督下由 AI 代理操作钱包这一新兴需求。分层安全模型（合约白名单、方法门控、美元阈值、每日限额）提供了可配置的纵深防御，可根据每个部署的风险承受能力进行调整。

跨 EVM 和 Solana 生态系统的多链支持，结合运行时链扩展能力，确保钱包在区块链格局演变中保持适用性。自托管架构避免了商业 MPC 解决方案的托管信任假设，同时通过开放的、经过充分研究的密码学协议提供可比的安全保障。
